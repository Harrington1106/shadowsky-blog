# 全站公开页 UI 重整美化 Spec

## Why
当前公开页已经历过多轮样式调整，仓库中的页面结构、局部样式和交互脚本与旧版美化计划不再完全一致，继续沿用旧计划会造成设计目标与当前代码现状脱节。需要基于当前文件重新制定一份“先收敛布局与信息层级，再统一视觉语言”的公开页美化规格，优先解决页面看起来杂乱、频道骨架不统一、公共层复用不充分的问题。

## What Changes
- 以当前仓库中的公开页实际结构为基线，重新定义全站公开页的 UI 美化目标与实施边界。
- 统一首页、博客、文章详情、瞬间、收藏、RSS、ACG、关于页、404 页等公开页的导航、页脚、主容器、区块标题和工具条设计语言。
- 重新梳理各频道页面的首屏骨架与信息层级，优先解决布局节奏混乱、内容主次不清、模块堆叠无序的问题。
- 收敛重复的页内样式、重复初始化脚本与重复交互结构，将可复用的视觉层和交互层继续沉淀到公共样式与公共脚本。
- 规范液态玻璃、模糊、阴影、渐变和动画的使用强度，避免装饰强于内容，并降低前台页面的性能压力。
- 统一加载态、空状态、按钮文案与提示语，减少公开页中英文混用和页面间反馈风格分叉。
- 明确本轮改造仅面向公开页 UI 与前端结构整理，不包含 `admin/`、后端接口、数据模型或业务规则重写。

## Impact
- Affected specs: 公开页视觉系统、页面骨架规范、导航与工具条规范、状态反馈规范、公共样式复用规范
- Affected code: `css/style.css`、`css/common-animations.css`、`js/main.js`、`index.html`、`blog.html`、`post.html`、`moments.html`、`bookmarks.html`、`rss.html`、`acg.html`、`anime.html`、`manga.html`、`edits.html`、`about.html`、`404.html`、相关页面脚本与开发文档
- Affected docs: `.trae/specs/realign-public-page-polish/spec.md`、相关 `tasks/checklist`、`docs/CODE_WIKI.md` 与后续公开页设计/实现说明

## 当前公开页现状与分组策略
### 逐页现状
- `index.html`：公开站点门户页，承担作者身份、站点气质和频道入口职责，但当前更偏视觉开场页，入口分发能力需要增强。
- `blog.html`：文章列表与发现页，已具备搜索、分页和多视图能力，但信息层级仍容易退化为单纯列表页。
- `post.html`：长文阅读页，承担正文、目录、相关推荐与辅助信息承载职责，可读性和稳定阅读面板优先级最高。
- `moments.html`：生活流/画册页，兼具图片流、统计卡、热力图和筛选操作，当前需要重新划分主次层级。
- `bookmarks.html`：资源目录与检索页，承担搜索、分类和收藏卡展示，需要从工具页提升为策展页。
- `rss.html`：订阅阅读工作台，已经是典型功能型页面，重点是明确三栏分工和阅读优先级。
- `acg.html`：二次元内容聚合入口页，承担番剧、漫画、视频入口聚合职责，属于频道入口型边缘页。
- `anime.html`：番剧列表页，结构更接近媒体列表/筛选页，适合跟随统一列表骨架收敛。
- `manga.html`：漫画列表页，与 `anime.html` 同属媒体列表骨架，优先跟随公共层而非单独发散。
- `edits.html`：剪辑/视频页，属于内容列表与播放器入口组合页，需要跟随媒体频道样式语言。
- `about.html`：个人介绍页，承担品牌说明和自我介绍职责，属于品牌说明型边缘页。
- `404.html`：错误引导页，承担兜底跳转和情绪缓冲职责，属于状态反馈型边缘页。

### 页面分组策略
- 核心六页：`index / blog / post / moments / bookmarks / rss`，作为本轮公开页重整的主战场，允许针对频道特性做明确的信息架构调整和模块重排。
- 边缘入口页：`acg / about / 404`，跟随核心页沉淀出的导航、页脚、容器、标题、状态区和交互反馈规范，不单独发明新的壳层语言。
- 边缘列表页：`anime / manga / edits`，跟随博客/收藏/瞬间页沉淀出的列表页骨架、筛选工具条、卡片密度和状态区规范，维持频道差异但避免新的结构分叉。
- 非公开页与非 UI 范围：`admin/`、后端接口、数据模型、业务规则、内容抓取流程不纳入本轮范围。

### 分组执行原则
- 核心六页优先解决“首屏骨架、主次层级、区块节奏、阅读路径”问题，再做视觉细化。
- 边缘页优先复用公共层和核心页模式，只在频道信息结构确有差异时保留最小私有实现。
- 若边缘页与核心页共用同类布局，优先复用同一套容器、工具条、卡片与状态区类名，不为局部页面新增平行体系。

### 公共壳层现状差异与收敛方向明细
#### 导航
- 现状差异：12 个公开页基本都保留 `#navbar`、站点入口、桌面导航链接、主题切换和移动端菜单按钮，但当前页高亮、品牌标题和背景强度仍由各页面手写控制；`index.html` 更接近透明悬浮首页导航，`blog / moments / bookmarks / rss / about / 404` 等页多为带边框与模糊的固定导航，`acg / anime / manga / edits` 还引入“二次元空间”品牌标题，导致同一站点内存在“总站导航”和“频道导航”两套观感。
- 收敛方向：保留统一的 `#navbar`、主题切换、菜单按钮和链接顺序，抽出单一公共导航壳层；通过页面级数据标记或配置声明当前页高亮、品牌标题、副标题和透明度等级，而不是继续在每个页面复制整段导航 DOM 与样式类；允许 `index / acg / 404` 使用有限的变体，但变体只能改品牌文案与装饰等级，不改结构骨架。

#### 页脚
- 现状差异：`blog / moments / bookmarks / post / rss / acg / anime / manga / edits / about` 已有近似页脚，但背景、边框、字号、间距、访问计数器样式仍存在分叉；`index.html` 使用首屏内部的绝对定位页脚而不是标准 `footer`；`404.html` 使用 `ShadowQuake` 与动态年份，但未复用访问计数器；`about.html` 在正文区额外插入一段“Designed with”文案，形成双重页尾表达。
- 收敛方向：定义“标准页脚”“沉浸首屏页脚”“状态页页脚”三种受控变体，并统一版权、访问计数、分隔符、年份策略和主题表现；除首页与 404 的必要特例外，其余公开页应复用同一 `footer` 结构与计数胶囊，不再各页微调一套背景和边框组合。

#### 工具条
- 现状差异：工具条形态已经明显分叉，`blog.html` 将搜索框和视图切换拆成两段；`moments.html` 采用粘性筛选条并叠加搜索、标签和随机按钮；`bookmarks.html` 把搜索区与分类导航拆成上下两层；`rss.html` 把刷新、订阅源、设置按钮分散到三栏头部；`anime / manga` 使用纯 pill 过滤组；`edits.html` 由脚本生成分类过滤容器，尚未接入统一工具条语义。
- 收敛方向：建立“发现型工具条”“筛选型工具条”“工作台头部工具条”三类公共模式，统一输入框高度、按钮尺寸、胶囊筛选器、粘性偏移、滚动阴影和断点折叠规则；页面只保留频道专属控件组合，不再各自维护不同的容器、留白和交互反馈。

#### 卡片
- 现状差异：仓库已存在 `glass-card` 等公共类，但具体卡片家族仍混杂使用；`moments.html` 统计卡、热力图卡与图片卡风格不完全一致，`bookmarks.html` 的资源卡和空状态仍偏工具页，`blog.html` 的文章卡依赖列表脚本独立渲染，`acg / anime / manga / edits` 的媒体卡、视频卡和加载卡又形成另一套频道模板，`rss.html` 的订阅项与文章项则更接近工作台列表项而非通用卡片。
- 收敛方向：将公开页卡片收口为“摘要卡”“内容卡”“媒体卡”“列表项卡”“状态卡”五类基础组件，统一圆角、阴影、边框、hover、标题层级、元信息区和操作区；保留媒体比例、封面布局和阅读列表密度等频道特性，但不再让每个页面单独定义新的卡片语言。

#### 状态区
- 现状差异：加载、空状态和提示语目前缺少统一体系，`acg / anime / manga / edits` 普遍使用居中的旋转圆环，`rss.html` 有“准备阅读”的默认空状态，`bookmarks.html` 有“未找到相关结果”区块，`blog / post / index / about / anime / manga / edits / acg / rss / moments / bookmarks` 仍保留英文 `Loading...` 访问计数占位，状态文案语言和视觉容器不一致。
- 收敛方向：建立“加载中 / 无数据 / 无结果 / 错误 / 待选择”五类状态区规范，统一中文文案、图标语义、容器边距、占位尺寸和降级策略；将访问计数、列表加载占位、空结果区和工作台默认面板纳入同一状态反馈语言，避免英文占位和页面间语气割裂。

#### 移动菜单
- 现状差异：几乎所有公开页都复制了一整套 `#mobile-menu-btn + #mobile-menu` 标记，链接顺序大体一致，但文案存在“视频 / ACG”“总站 / 频道”差异，按钮和容器类名也有轻微偏移；移动端主题切换仍大量依赖内联 `onclick="toggleTheme()"`，导致菜单和主题交互都分散在页面模板里。
- 收敛方向：将移动菜单视为导航壳层的一部分统一输出，使用单一菜单模板和单一开关逻辑，通过配置切换高亮项与频道别名；继续保留 `#mobile-menu-btn` 与 `#mobile-menu` 作为兼容挂载点，但不再在每个页面复制整段菜单结构和交互绑定。

#### 页内样式
- 现状差异：所有主要公开页都保留页内 `<style>`，其中 `post.html` 包含大段排版、代码块和深色模式样式，`blog / moments / bookmarks / rss / acg / anime / manga / edits / about / 404 / index` 也分别保留字体、滚动条、Lenis、卡片 hover、动画延时或局部补丁；同时存在多处内联 `style="animation-delay: ..."`、`style="transform-style: preserve-3d;"` 等页内样式，公共视觉层尚未完全沉淀到共享样式表。
- 收敛方向：把字体基线、动画延迟工具类、滚动条隐藏、通用 hover、Lenis 辅助样式、常见状态卡与页脚/导航补丁逐步迁移到 `css/style.css` 或受控的页面级样式分区；仅保留正文排版、特殊画布、小游戏、专属视觉变量等无法公共化的最小私有样式。

#### 重复脚本初始化
- 现状差异：公开页普遍依赖 `js/main.js`，但仍在页内重复执行 `lucide.createIcons()`、`toggleTheme()`、Lenis 初始化、`pageshow` 恢复、`DOMContentLoaded` 里的 `MediaLoader` / `VideoLoader` 启动逻辑；`acg / anime / manga / edits / bookmarks / moments / post / about` 各自维护一份 Lenis 与 bfcache 恢复代码，`anime / manga / acg` 还各自写了相似的数据加载启动入口，存在重复初始化和双重绑定风险。
- 收敛方向：把图标重绘、主题切换、导航高亮、移动菜单、访问计数、可选平滑滚动和 bfcache 恢复纳入统一公共 bootstrap；页面专属模块改为通过数据属性或小型注册表声明初始化需求，例如声明 `media-loader`、`video-loader`、`reading-page`、`moments-page`，由公共入口按需拉起，避免重复执行和脚本职责漂移。

## ADDED Requirements
### Requirement: 基于现状重建公开页美化基线
系统 SHALL 以当前仓库中的公开页结构、公共样式和公共脚本为唯一事实基线，重新制定全站公开页美化方案，而不是直接沿用历史草案中的页面结构假设。

#### Scenario: 重新评估美化范围
- **WHEN** 开发者开始制定或执行公开页美化计划
- **THEN** 应先以当前 `html/css/js` 文件现状确认页面结构、公共层能力与页面差异
- **THEN** 历史设计文档只能作为参考，不得直接视为当前实现状态

### Requirement: 公开页布局与信息层级收敛
系统 SHALL 对全站公开页进行布局和信息层级收敛，确保每个页面都具备清晰的主次关系、稳定的区块节奏和一致的阅读路径。

#### Scenario: 用户在不同公开页之间切换
- **WHEN** 用户在首页、频道页、内容页和功能页之间切换
- **THEN** 应能感知到一致的导航入口、标题层级、主内容容器和区块节奏
- **THEN** 页面差异应主要体现在频道内容编排，而不是基础布局规则混乱

### Requirement: 逐页现状盘点与分组实施
系统 SHALL 在执行公开页重整前，先完成 12 个公开页的逐页现状盘点，并按照“核心六页优先、边缘页跟随”的方式组织实施顺序与约束。

#### Scenario: 规划公开页改造批次
- **WHEN** 开发者制定任务拆分、设计文档或实现计划
- **THEN** 应明确 `index / blog / post / moments / bookmarks / rss` 为核心六页
- **THEN** 应明确 `acg / anime / manga / edits / about / 404` 为边缘公开页并采用跟随式收敛策略
- **THEN** 不应再以“所有公开页同权并行重做”的方式分散实现重点

### Requirement: 核心六页方向明确
系统 SHALL 为核心六页分别给出清晰的信息架构方向，确保本轮改造优先解决这些页面的主目标和主阅读路径。

#### Scenario: 定义核心六页的页面方向
- **WHEN** 团队更新公开页重整方案
- **THEN** `index.html` 应被定义为“站点门户与频道入口页”，重点补足内容入口分发和首屏后续承接
- **THEN** `blog.html` 应被定义为“编辑式内容发现页”，重点强化摘要区、搜索/视图工具条和列表节奏
- **THEN** `post.html` 应被定义为“稳定深度阅读页”，重点保护正文可读性、目录侧栏和阅读面板秩序
- **THEN** `moments.html` 应被定义为“生活画册页”，重点前置图片流并下沉统计与热力图
- **THEN** `bookmarks.html` 应被定义为“策展型资源目录页”，重点增强精选入口、分类关系与空状态表达
- **THEN** `rss.html` 应被定义为“阅读工作台页”，重点明确订阅源、中栏收件箱和右栏沉浸阅读器的分工

### Requirement: 边缘页跟随核心规范
系统 SHALL 要求边缘公开页跟随核心页沉淀出的公共壳层、列表模式和状态反馈规范，避免形成新的视觉与结构分叉。

#### Scenario: 调整边缘公开页
- **WHEN** 开发者处理 `acg / anime / manga / edits / about / 404`
- **THEN** `acg / about / 404` 应优先跟随门户页、品牌说明页与状态反馈页的公共规则
- **THEN** `anime / manga / edits` 应优先跟随列表页与媒体页的工具条、卡片栅格、筛选区和状态区规则
- **THEN** 边缘页仅可保留频道专有模块，不应新增与核心页平行的导航、容器或反馈体系

### Requirement: 公共 UI 壳层统一
系统 SHALL 为公开页统一导航、页脚、主容器、工具条、卡片、状态区和移动端菜单的公共 UI 壳层，减少重复实现和局部风格分叉。

#### Scenario: 页面复用公共结构
- **WHEN** 任一公开页需要渲染导航、工具区、卡片区、状态区或底部信息
- **THEN** 应优先复用公共样式类和公共脚本逻辑
- **THEN** 不应在多个页面中继续复制同一套结构和仅有细微差异的样式片段

#### Scenario: 对照公共壳层差异明细执行收敛
- **WHEN** 开发者处理导航、页脚、工具条、卡片、状态区或移动端菜单
- **THEN** 应以“先保留公共挂载点和链接顺序，再抽离样式与初始化差异”为原则推进
- **THEN** `index / 404 / acg` 等特例页只能保留受控变体，不应继续形成新的壳层分支
- **THEN** 页面的私有实现应主要体现在频道文案、信息编排和少量专属控件，而不是基础壳层结构

### Requirement: 受控视觉效果
系统 SHALL 对液态玻璃、背景模糊、阴影、渐变和动效做分级控制，保证视觉统一的同时不压制正文与内容卡片的可读性。

#### Scenario: 页面展示视觉装饰
- **WHEN** 页面使用玻璃态、模糊层、发光、悬浮阴影或持续动画
- **THEN** 这些效果应主要服务于导航、工具条、摘要面板和浮层，而非大面积覆盖正文阅读区
- **THEN** 页面不应因高强度模糊或装饰动画产生明显的阅读干扰或性能压力

### Requirement: 响应式一致性约束
系统 SHALL 为公开页重整补充明确的响应式规则，确保核心六页和边缘页在手机、平板、桌面场景中保持一致的阅读路径和可操作性。

#### Scenario: 在不同断点查看公开页
- **WHEN** 开发者验证 `375 / 768 / 1024 / 1440` 等主要断点
- **THEN** 页面不应出现横向滚动、导航遮挡正文、工具条溢出或卡片密度失衡
- **THEN** 核心六页的首屏骨架、操作入口和状态区应保持清晰且可触达
- **THEN** 边缘页应跟随相同断点策略，而不是单独定义与核心页冲突的收缩方式

### Requirement: 文档同步要求
系统 SHALL 在每次公开页重整方案或实现边界发生变化后，同步更新规格、任务清单、代码索引和相关设计说明，确保文档与代码现状一致。

#### Scenario: 公开页方案被补充或调整
- **WHEN** 团队新增页面方向、分组策略、验收条件或实现边界
- **THEN** 应同步更新 `.trae/specs/realign-public-page-polish/` 下的 `spec / tasks / checklist`
- **THEN** 若页面职责、脚本归属或共享壳层说明发生变化，应同步更新 `docs/CODE_WIKI.md` 和相关公开页文档
- **THEN** 文档描述不得与仓库现状冲突，也不得虚构当前仓库中不存在的开发文档或项目状态文件

### Requirement: 兼容性约束
系统 SHALL 在公开页重整过程中保持现有页面功能挂载点、主题能力和主流浏览环境下的可用性，避免 UI 改造引发功能回归。

#### Scenario: 执行结构与样式收敛
- **WHEN** 开发者调整公开页 HTML、CSS 或公共脚本
- **THEN** 应保留现有关键 ID、数据容器、脚本挂载点和导航行为，避免破坏 `js/main.js` 及页面专属脚本的现有依赖
- **THEN** 应兼容浅色/深色主题、键盘焦点、移动端菜单与 `prefers-reduced-motion`
- **THEN** 对 `backdrop-filter`、模糊、阴影和动画应提供可降级表现，确保在不完全支持相关视觉特性的环境中仍可正常阅读与操作

### Requirement: 公开页文案与状态反馈统一
系统 SHALL 统一公开页的加载态、空状态、错误提示、按钮文案和辅助说明语言，确保站点前台在中文语境下保持一致的反馈体验。

#### Scenario: 页面加载或无结果
- **WHEN** 公开页处于加载、空数据、筛选无结果或异常提示场景
- **THEN** 应提供统一语气、统一语言和统一视觉层级的反馈
- **THEN** 不应继续出现分散的英文占位文案和明显不一致的状态区样式

### Requirement: 页面私有实现继续收敛
系统 SHALL 逐步收敛公开页中的页内样式、重复脚本初始化和重复模板片段，提升页面结构与样式的可维护性。

#### Scenario: 维护公开页代码
- **WHEN** 开发者调整公开页视觉或结构
- **THEN** 应优先将可复用样式、交互初始化和模板骨架抽到公共层
- **THEN** 页面私有代码只保留频道特有布局、数据容器和少量无法抽象的局部逻辑

#### Scenario: 处理页内样式与重复初始化
- **WHEN** 开发者发现页内 `<style>`、内联动画延迟、重复 `lucide.createIcons()`、Lenis 启动或相似的 `DOMContentLoaded` 初始化
- **THEN** 应优先评估是否可迁移到 `css/style.css`、`js/main.js` 或统一的页面模块注册入口
- **THEN** 若暂时无法抽离，应在文档中明确其为“保留的页面私有实现”而不是默认继续复制到其他页面

## MODIFIED Requirements
### Requirement: 全站液态玻璃视觉系统的适用方式
现有公开页液态玻璃系统 SHALL 继续作为公共视觉基础层存在，但其职责应调整为“提供统一的视觉令牌和组件能力”，而不是替代页面级信息架构整理；公开页美化应优先解决布局、层级、文案和复用问题，再在此基础上控制玻璃效果强度。

#### Scenario: 使用液态玻璃公共层
- **WHEN** 页面继续复用现有液态玻璃类和视觉变量
- **THEN** 应将其作为统一基础能力使用
- **THEN** 不应仅靠增加玻璃效果来掩盖页面布局混乱或模块关系不清的问题

### Requirement: 旧版美化草案的使用方式
已有的博客专项美化草案和核心页面美化草案 SHALL 视为历史参考输入，其未落地或与当前文件不一致的部分，需要在新的公开页美化规格中重新确认后再采用。

#### Scenario: 对照历史草案
- **WHEN** 开发者参考旧版 `spec`、`design` 或 `plan`
- **THEN** 应先核对当前文件是否仍保留对应结构与命名
- **THEN** 若不一致，应以新的公开页重整规格为准
