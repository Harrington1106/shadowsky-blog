# 首页单屏封面与背景统一设计文档

## 概述
- 项目：`shadowsky-blog`
- 日期：`2026-06-11`
- 主题：首页单屏封面化与顶部背景断层修复
- 范围：`index.html`、`css/style.css`
- 页面目标：首页变成真正的一屏封面页，所有元素首屏内完整显示，且背景从页面顶部到底部连续统一

## 设计前提
- 当前首页已经删除标签区与 `home-gateway` 入口块，保留了导航、头像、主标题、副句与底栏。
- 当前问题已从“内容过多”转向“封面构图不够严格”：首屏元素接近封面化，但页面仍然存在滚动空间，顶部背景也有断层感。
- 用户要求不是“更接近一屏”，而是“整个首页所有内容都在一屏内显示完”。

## 用户确认结论
- 采用方案 A：单屏封面 + 背景全屏统一。
- 首页应成为严格的一屏页。
- 顶部背景必须覆盖完整，不能再出现导航上方背景没铺干净的突兀区域。
- 本次工作仍然只聚焦首页，不扩散到其他页面。

## 当前问题归纳

### 1. 首页还不是真正的单屏
- 虽然首页内容已经减少，但当前 hero、导航和底栏的垂直分配仍不够严格。
- 结果是首页视觉上像封面，但布局上仍然保留了页面感和滚动感。

### 2. 顶部背景断层明显
- 当前背景主层更多集中在 hero 区。
- 导航上方存在一块视觉上不连贯的区域，像背景没有完整覆盖到页面顶部。
- 这会让导航和首屏背景看起来像两层不同的界面拼接在一起。

### 3. 封面语言仍然不够“整体”
- 导航、hero、底栏在视觉上已接近统一，但在高度分配和背景归属上仍然像三个单独区域。
- 高级封面应当像一张完整海报，而不是三段堆叠。

## 目标
- 首页所有元素在桌面视口内一屏显示完。
- 导航、hero、底栏全部纳入同一个首屏封面系统。
- 背景从页面顶部到底部连续存在，不再出现顶部断层。
- 保留现有梦幻科技感，但整体更完整、更克制。

## 非目标
- 不恢复内容入口区。
- 不新增新的首页模块。
- 不改导航交互逻辑和移动菜单逻辑。
- 不改粒子脚本行为，只调整背景承载方式和容器关系。
- 不改其他公开页。

## 方案比较

### 方案 A：单屏封面 + 背景全屏统一（采用）
- 做法：首页严格按 `100dvh` 构图，导航、hero、底栏全部位于同一张背景上，页面不再依赖额外滚动空间。
- 优点：最彻底，能够同时解决“页面超出一屏”和“顶部背景断层”。
- 缺点：对 hero 间距、底栏占位和顶部安全区的控制要求更高。

### 方案 B：单屏封面 + 顶部补背景层
- 做法：保留现有布局，仅补顶部背景覆盖。
- 优点：改动较小。
- 缺点：更像修补，无法彻底消除封面不完整感。

### 方案 C：继续压缩元素，不改整体结构
- 做法：只继续缩头像、缩间距、缩底栏。
- 优点：实现最轻。
- 缺点：不能保证真正一屏，也难以根治背景断层。

## 采用方案
- 采用方案 A：单屏封面 + 背景全屏统一。

## 页面结构设计

### 1. 根容器
- 首页根容器改为严格以视口高度为基准。
- 首屏内容容器不再只是“最小高度接近一屏”，而是作为真正的封面容器承载全部元素。
- 页面在首页场景下应避免额外纵向滚动。

### 2. 导航
- 导航继续固定在顶部并保留液态玻璃样式。
- 但导航不再像悬浮在另一层背景上，而是明确浮在首页完整背景之上。
- 导航上方不允许露出任何与封面背景不一致的底色区域。

### 3. Hero 主区
- hero 主区承担头像、主标题、副句。
- hero 的高度由视口剩余空间分配，而不是靠较松的 `pt` / `pb` 组合撑开。
- 头像、标题、副句会继续采用更紧凑的垂直节奏。

### 4. 底栏
- 底栏保留。
- 底栏在封面内承担“签名信息”的角色，而不是页面附加内容。
- 底栏位置会被收进同一屏，不再把首页总高度继续往下撑。

## 背景设计

### 1. 背景整体性
- 首页背景必须从页面最顶部开始连续存在。
- 背景不再只属于 hero 区，而是属于整个首页封面容器。
- 导航、hero、底栏共同叠加在同一套背景系统之上。

### 2. 顶部断层修复
- 断层修复的原则不是单独补一条颜色或遮罩。
- 真正的修复方式是统一背景承载关系，让页面顶部、本体内容区和底部信息区都被同一背景覆盖。
- 最终效果应是浏览器顶部到页面底部像同一张完整封面。

### 3. 背景视觉强度
- 保留当前粒子网络和柔色光斑。
- 背景仍然是氛围层，不应成为抢夺注意力的主视觉层。
- 顶部覆盖补齐后，背景需要显得完整，但不能更吵。

## 布局策略

### 1. 单屏策略
- 首页在桌面端采用严格单屏布局。
- 导航、hero、底栏三部分的总高度必须被控制在 `100dvh` 内。
- 需要避免底栏被挤出视口或 hero 被压得过于拥挤。

### 2. 垂直分配
- 顶部：导航与必要安全间距
- 中部：hero 主区占据主要视觉空间
- 底部：底栏作为封面下缘信息
- 三者关系应稳定，避免依赖过大的外边距和补丁式负空间。

### 3. 移动端策略
- 目标仍然是封面感优先。
- 若移动端极小高度下出现无法兼顾可读性与完整显示的冲突，则以不变形、不重叠、不裁切为优先。
- 但桌面端必须先保证严格一屏。

## 实施边界
- 只修改：
  - `index.html`
  - `css/style.css`
  - `js/core-pages-ui.test.js`
  - 必要时 `docs/CODE_WIKI.md`
- 不修改：
  - 导航功能
  - 粒子逻辑
  - 右键菜单逻辑
  - 其他页面布局

## 验收标准
- 首页在桌面端打开后，所有元素首屏内完整可见。
- 首页不再需要滚动才能看全导航、hero 和底栏。
- 顶部背景覆盖完整，不再有突兀断层。
- 导航、hero、底栏属于同一张完整封面，而不是三层拼接。
- 页面整体更像封面海报，而不是普通落地页。

## 风险与控制
- 风险：为了强制单屏而把 hero 压得过紧
  - 控制：优先调整整体高度分配，而不是简单暴力减少字号和可读性
- 风险：顶部补背景后视觉变脏
  - 控制：统一背景承载层，而不是局部加补丁遮罩
- 风险：移动端极小高度下内容拥挤
  - 控制：桌面端先严格达标，移动端以不重叠和不裁切为底线

## QA 审计报告

| 隐患点 | 严重程度 | 修复建议 |
|---|---|---|
| 单屏约束可能导致小屏高度紧张 | 中 | 优先做整体高度分配，不靠局部强压字号解决 |
| 背景顶部补齐若采用补丁式做法，后续易再次分层 | 中 | 必须从容器关系上统一背景承载 |
| 本次不涉及 PHP 接口、SQL 路径 | 低 | 无新增 SQL 注入风险 |
| 本次不涉及 Admin 认证流程 | 低 | 无新增 Token 校验风险 |
| 粒子背景覆盖整页后可能增加首屏绘制压力 | 低 | 保持背景为氛围层，不新增更重的动态特效 |

QA 结论：方案为低风险，可进入实施计划阶段。

## 结论
本次首页不再只是“优化首屏元素比例”，而是把首页彻底收口为一张完整的单屏封面。通过严格控制视口高度分配，并统一导航、hero 与底栏的背景归属关系，首页将同时解决“内容超出一屏”和“顶部背景断层”两个问题，获得更完整、更高级的整体感。
