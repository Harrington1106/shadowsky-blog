# 全站公开页 UI 布局修复设计文档

## 概述
- 项目：`shadowsky-blog`
- 日期：`2026-06-11`
- 主题：前台全部公开页面 UI 布局异常排查与恢复
- 范围：`index.html`、`blog.html`、`post.html`、`moments.html`、`bookmarks.html`、`rss.html`、`acg.html`、`anime.html`、`manga.html`、`edits.html`、`about.html`、`404.html`
- 不包含：`admin/` 后台、后端接口逻辑、数据结构改造

## 文档前提
- 仓库内未发现用户协议中提到的 `docs/DEVELOPMENT_MANUAL.md` 与 `project-state.yml`。
- 本设计以当前仓库代码、`docs/CODE_WIKI.md`、现有公开页样式层与页面结构为准。
- 当前仓库已存在一轮“核心公开页 UI 美化”设计，本次设计不重复讨论美化方向，而是聚焦“布局错乱排查与恢复”。

## 背景
当前前台公开页已经接入一套共享的液态玻璃样式系统，并在多个页面引入了 `public-shell`、`section-shell`、`utility-glass-bar`、`public-footer` 等公共类。但全站仍存在以下结构性风险：

- 公开页接入公共骨架的深度不一致，部分页面仍保留大量独立容器和局部样式。
- 固定导航、粘性工具条、局部滚动容器和动态内容区域没有统一的空间规则。
- ACG 系列、关于页和 404 页仍保留较多独立实现，容易在断点切换时出现宽度、间距和层级分叉。
- 页面之间页脚、主容器、首屏偏移、工具条留白、卡片栅格节奏不完全一致。

本次目标不是追求新一轮视觉重构，而是以最小必要改动恢复全站布局稳定性，让所有公开页在桌面端、平板端和手机端都能稳定展示。

## 目标
- 全面排查全部公开页的布局异常和潜在错乱点。
- 统一恢复公开页容器、工具条、卡片、页脚和首屏偏移规则。
- 修复响应式适配失效、元素错位、间距失调、组件堆叠错乱与横向溢出。
- 建立可复用的全站布局钩子和验证方式，降低后续页面再次分叉的概率。

## 非目标
- 不改 `admin/` 后台视觉和布局。
- 不重写业务逻辑、数据请求流程和接口结构。
- 不引入新的大型前端依赖。
- 不将本次任务扩大为整站重新设计。

## 修复策略选择

### 备选方案
- 方案 A：逐页局部打补丁
  - 优点：改动小，实施速度快。
  - 缺点：容易遗留公共样式分叉，后续维护成本高。
- 方案 B：统一公共壳层 + 全页面补齐结构钩子 + 分页做响应式收口
  - 优点：改动可控，能同时解决共享问题和页面差异问题，适合全站修复。
  - 缺点：需要先整理公共层，再逐页收口。
- 方案 C：全站重新搭建布局骨架
  - 优点：统一度最高。
  - 缺点：改动面过大，业务回归风险高，不符合当前“恢复正常布局”的目标。

### 采用方案
- 采用方案 B。
- 原因：本次问题是全站公开页布局稳定性问题，而不是单页孤立问题；需要以共享样式层为中心先做统一，再逐页补齐差异。

## 全站修复架构

### 1. 公共层统一
- 将导航顶部偏移、主容器宽度、页脚内容宽度、工具条外边距、卡片栅格间距和交互态收敛到 `css/style.css`。
- 减少页面级 `max-w-*`、`mx-auto`、`px-*` 与公共容器重复叠加。
- 为固定导航和页面主内容建立统一的顶部安全间距，避免首屏被遮挡。

### 2. 结构层补钩子
- 全站统一使用少量语义钩子，避免再依赖分散的局部类名维持布局：
  - `data-ui-shell`
  - `data-ui-toolbar`
  - `data-ui-section`
  - `data-ui-panel`
  - `data-ui-group`
- 钩子只承担结构语义，不改变现有业务容器 ID 和 JS 绑定点。

### 3. 页面层收口
- 每个页面只处理与当前布局异常直接相关的结构和样式。
- 优先修复：
  - 容器宽度冲突
  - 首屏偏移和 sticky 冲突
  - 栅格与列数跳变
  - 页脚分叉
  - 动态内容加载后的溢出和堆叠

## 布局异常排查清单

### 首页 `index.html`
- 风险点：
  - Hero 区高度较大，背景 blob、画布和底部信息条共存，手机端容易造成视觉重叠。
  - 首屏底部 `public-footer` 采用绝对定位，需确认与内容区和视口高度的关系稳定。
  - 首页入口摘要区新增后，需检查首屏结束后的留白节奏是否与导航偏移一致。
- 重点表现：
  - 首屏底部信息遮挡
  - 内容入口区与首屏底部挤压
  - 小屏文本换行异常

### 博客页 `blog.html`
- 风险点：
  - 搜索区、视图切换器和博客摘要带均为工具层，平板端最容易拥挤换行。
  - 博客列表容器和工具条之间节奏依赖多个局部类。
- 重点表现：
  - 搜索栏与视图切换器换行错位
  - 摘要带与文章列表间距不稳定
  - 固定导航下首屏留白不一致

### 文章页 `post.html`
- 风险点：
  - 固定导航、头图 hero、正文阅读面板、桌面侧栏、移动抽屉同时存在。
  - 正文区与侧栏比例依赖断点切换，若容器宽度不稳容易导致正文挤压。
- 重点表现：
  - 正文与侧栏列宽不平衡
  - 移动端目录抽屉和悬浮按钮遮挡
  - 头图结束后内容顶部节奏跳变

### 随手拍 `moments.html`
- 风险点：
  - sticky 筛选栏、热力图横向区域、瀑布流列布局、灯箱详情面板叠加复杂。
  - 列布局与响应式切换结合时最容易横向溢出。
- 重点表现：
  - sticky 工具条遮挡内容
  - 热力图区域横向滚动不受控
  - 瀑布流列数切换时卡片错位
  - 灯箱在小屏布局拥挤

### 收藏页 `bookmarks.html`
- 风险点：
  - 搜索框、精选摘要区、分类导航、动态分类列表连续堆叠。
  - 分类导航 sticky 后若顶部偏移不统一，容易与固定导航冲突。
- 重点表现：
  - 分类导航吸顶位置错误
  - 搜索区和精选区垂直节奏紊乱
  - 结果为空时布局留白不自然

### RSS 页 `rss.html`
- 风险点：
  - 三栏布局建立在 `h-screen + flex + overflow-hidden` 上，最依赖滚动上下文正确性。
  - 左栏、中栏、阅读区在不同断点的隐藏/显示关系复杂。
- 重点表现：
  - 内容区不可见或被裁切
  - 三栏宽度失衡
  - 小屏下工具栏或列表区无法正常滚动

### ACG 首页 `acg.html`
- 风险点：
  - 仍保留 `max-w-7xl` 与 `public-shell` 混用。
  - 多个频道区块采用独立间距规则，容易造成段落节奏分叉。
- 重点表现：
  - 频道区块间距不统一
  - 栅格列数跳变不稳定
  - 页脚未完全接入 `public-footer`

### 追番页 `anime.html`
- 风险点：
  - 返回区、筛选器和卡片栅格依赖局部样式，平板端按钮数较多。
  - 仍保留 `max-w-7xl` 与公共容器叠加。
- 重点表现：
  - 筛选按钮换行挤压
  - 栅格列数切换后间距突变
  - 页脚与公共页不一致

### 漫画页 `manga.html`
- 风险点：
  - 与追番页结构相似，但脚本和页脚接入仍有差异。
- 重点表现：
  - 筛选工具条溢出
  - 内容栅格列数和留白不统一
  - 页脚、主内容边界与全站不一致

### 剪辑页 `edits.html`
- 风险点：
  - 分类工具条采用横向滚动容器，最容易在手机端遮挡或宽度超出。
  - 页脚仍为旧版实现。
- 重点表现：
  - 分类工具条滚动区域与遮罩表现异常
  - 视频卡片切列后节奏不稳定
  - 页脚与容器宽度分叉

### 关于页 `about.html`
- 风险点：
  - 主体内容已有 `public-shell`，但仍叠加了局部 `max-w-*`、独立页脚和多段自定义结构。
  - 社交链接区数量较多，手机端最容易过密。
- 重点表现：
  - 主体宽度与公共壳层叠加
  - 页脚仍走旧版容器
  - 卡片区和联系区间距不统一

### 404 页 `404.html`
- 风险点：
  - 导航和主体已经部分接入公共类，但页脚仍保留独立实现。
  - 游戏区域为特殊布局，容易与按钮区、页脚形成节奏断层。
- 重点表现：
  - 错误页主体与页脚风格分叉
  - 游戏面板与返回按钮纵向间距异常
  - 导航与主体顶部偏移不完全一致

## 根因分类

### 1. CSS 样式冲突
- 公共类与页面局部类并存，出现宽度、内边距、边框和背景不一致。
- `public-shell`、`public-footer`、`utility-glass-bar` 已存在，但部分页面仍沿用旧样式实现。

### 2. Flex / Grid 参数不一致
- 各页面使用的 `grid-cols-*`、`gap-*`、`columns-*` 和 `flex-1` 组合缺少统一规范。
- 同类页面在不同断点使用不同列数和外边距，导致视觉节奏不统一。

### 3. 媒体查询和断点收口不足
- 页面依赖 Tailwind 响应式类，但未形成“断点级布局规则”。
- 平板尺寸是最容易暴露问题的区间，常见于工具条换行和内容区压缩。

### 4. 盒模型与宽度计算问题
- 固定导航、阴影、边框与 `calc(100% - x)` 混用时，会放大容器溢出和错位。
- `max-w-*` 与 `public-shell` 同时存在，导致有效内容宽度不可预期。

### 5. 第三方组件与动态内容污染
- Lucide 图标替换、动态卡片插入、热力图与视频内容渲染后，会暴露结构不稳定问题。
- 动态内容常常在空状态正常、加载完成后错位。

### 6. 视口和滚动上下文异常
- `Lenis`、`overflow-hidden`、`sticky`、`fixed` 共存页面对滚动上下文要求更高。
- `rss.html`、`moments.html`、`post.html` 是高风险页。

## 实施设计

### 共享样式层
- 在 `css/style.css` 中补充全站布局恢复层，统一：
  - 导航与正文顶部安全间距
  - 页面主容器最大宽度与横向留白
  - 页脚容器与页脚背景语义
  - 工具条换行与纵向间距
  - 卡片列表的标准 gap 规则
  - 常见小屏溢出防护

### 页面结构层
- 对所有公开页补齐或统一：
  - 主体 `data-ui-shell`
  - 工具条 `data-ui-toolbar`
  - 核心内容区 `data-ui-section`
  - 重要局部区域 `data-ui-panel`
- 保持现有业务 ID、脚本初始化点和动态容器不变。

### 页面收口顺序
1. 统一 `css/style.css` 中的公共布局恢复层。
2. 收口 `public-footer` 与主容器接入不一致的页面。
3. 修复高复杂度页面：`rss.html`、`moments.html`、`post.html`。
4. 修复 ACG 系列页面：`acg.html`、`anime.html`、`manga.html`、`edits.html`。
5. 修复静态页：`about.html`、`404.html`。
6. 回头统一小屏断点、空状态、加载态与页脚节奏。

- Chrome：主验证基线

### 验收标准
- 无横向滚动条
- 无导航遮挡核心内容
- 无工具条挤压、错位或越界
- 无栅格卡片重叠和异常留白
- 无页脚宽度和层级分叉
- 动态内容注入后布局仍稳定
- 主要交互区 focus 态可见

## 风险与控制
- 风险：统一公共层后影响已有页面个性
  - 控制：只收口布局骨架，不改变页面主题内容和主要视觉表达
- 风险：结构调整影响现有 JS 查询
  - 控制：保留原有 ID、数据容器和初始化入口
- 风险：玻璃效果叠加影响滚动性能
  - 控制：大面积内容区使用轻背景，重模糊只保留在导航、工具条、抽屉和弹层
- 风险：全站修改面较大，易遗漏边角页
  - 控制：以 12 个公开页为完整清单逐项验收

## QA 审计关注点

### 审计范围说明
- 本次设计主要作用于前端公开页布局层，不引入新的后端接口、SQL 写路径或管理端鉴权变更。
- 审计重点转为前端布局稳定性与性能，但仍保留用户协议要求的安全检查视角。

## 审计报告

| 隐患点 | 严重程度 | 修复建议 |
|---|---|---|
| 公开页公共壳层接入不一致，导致断点切换时宽度和间距分叉 | 中 | 统一使用 `public-shell`、`public-footer`、`utility-glass-bar` 并移除重复容器宽度定义 |
| `rss.html`、`moments.html`、`post.html` 中固定/粘性/局部滚动组合复杂，易出现内容不可见或遮挡 | 中 | 统一滚动上下文与顶部偏移规则，逐页验证 sticky/fixed 场景 |
| ACG 系列页仍保留独立页脚和 `max-w-*` 规则，响应式错位风险较高 | 中 | 把 ACG 系列页收口到共享页脚与统一主容器体系 |
| 大面积 `backdrop-filter` 与阴影在低性能设备可能造成滚动卡顿 | 低 | 仅在导航、工具条、抽屉和弹层启用强玻璃效果 |
| 本次任务未触碰 PHP SQL 写逻辑，当前未新增 SQL 注入面 | 低 | 后续后端改动仍需坚持预处理与输入校验 |
| 本次任务未触碰 Admin Token 流程，当前未新增后台鉴权变更面 | 低 | 后续管理端改动继续统一走现有 token 校验路径 |

QA 结论：当前设计方案为低风险，可进入实施计划阶段。

## 结论
本次全站公开页布局修复采用“统一公共壳层 + 全页面补齐结构钩子 + 分页做响应式收口”的方案，在不重写业务逻辑的前提下，优先解决容器宽度分叉、首屏偏移不一致、工具条错位、栅格异常和页脚分裂问题。完成后，公开页将具备更稳定的桌面端、平板端和手机端布局表现，并建立可复用的全站布局验证基线。
