## 一、问题：为什么白天飞起，晚高峰跪了？

如果你有一台海外的廉价 VPS（RackNerd、CloudCone、AWS Lightsail 等），你一定体验过这种割裂感：

```
白天 14:00 → YouTube 4K 秒开，Speedtest 跑满 100Mbps
晚上 21:00 → 720p 缓冲半天，Speedtest 不到 5Mbps
```

大多数人的第一反应是："晚高峰带宽不够了"。但真相远比这个复杂。

---

## 二、根因分析：TCP 在丢包场景下的崩溃

让我们做一些简单的计算。

### 2.1 带宽和丢包的关系

TCP 的吞吐量有一个著名的理论上限公式：

```
Throughput ≤ MSS / (RTT × √p)
```

其中：
- **MSS** = 最大报文段长度（通常 1460 字节）
- **RTT** = 往返时延（中美之间约 150ms）
- **p** = 丢包率

代入实际数据：

| 丢包率 p | TCP 理论最大吞吐量 |
|----------|-------------------|
| 0.1% | ~150 Mbps |
| 1% | ~50 Mbps |
| **5%** | **~22 Mbps** |
| **10%** | **~16 Mbps** |

> 注意：这是**理论最大值**。实际中，TCP 的拥塞控制（Cubic/BBR）在丢包时会主动降窗，实际吞吐往往只有理论值的一半。

### 2.2 晚高峰到底发生了什么？

我们用 `mtr` 做一个简单的测试。选择一台 RackNerd 洛杉矶机房的 VPS，在晚高峰 21:00 执行：

```bash
mtr -r -c 100 你的VPS_IP
```

典型输出：

```

1. 192.168.1.1                 0.0%   100    0.3   0.4   0.2   1.2   0.1
2. 10.74.0.1                   0.0%   100    3.2   3.5   2.8  15.3   1.8
3. 202.97.46.1                 0.0%   100    5.1   6.3   4.2  28.1   3.5
...
10. 219.158.33.1               4.0%   100  178.3 185.2 165.4 320.1  35.2
11. 219.158.97.130            12.0%   100  192.1 201.3 178.2 350.4  42.1  ← 丢包开始
12. 218.30.54.186             18.0%   100  195.2 210.4 182.3 380.5  55.3  ← 丢包加剧
```

**关键发现**：第 10-12 跳是 ChinaNet 163 骨干网的国际出口节点。这里的丢包率从 4% 飙升到 18%。这不是你 VPS 的问题——这是物理链路（163 骨干网）在晚高峰被挤爆了。

### 2.3 TCP 为什么会崩？

TCP 的核心假设是：**丢包 = 拥塞**。当发生丢包时，TCP 的拥塞控制算法会：

1. **减小拥塞窗口（cwnd）**：把发送速率砍半甚至降到 1 MSS
2. **触发重传**：等待 RTO（重传超时，通常 ≥200ms）才重发丢失的包
3. **慢启动恢复**：从极小窗口重新爬升

在晚高峰场景下，丢包率达到 10-20%，TCP 陷入一个死亡螺旋：

```
丢包 → 降速 → 重传 → 更多包排队 → 更大延迟 → 超时重传 → 进一步降速
```

用 Wireshark 抓包看，会发现大量的 `TCP Retransmission` 和 `TCP Dup ACK`，有效吞吐不到链路的 10%。

## 三、解法：不用 TCP 就行了

既然 TCP 在丢包场景下表现糟糕，我们换一个思路——**用 UDP**。

### 3.1 为什么 UDP 不怕丢包？

UDP 是无连接的。它没有拥塞窗口、没有重传机制、没有慢启动。你发多少它就送多少，不管路上的包丢了多少。

但纯 UDP 也有问题：不可靠。丢了的数据就是丢了。我们需要一个**基于 UDP 的可靠传输协议**——这就是 QUIC。

### 3.2 QUIC 和 Hysteria 2

**QUIC**（Quick UDP Internet Connections）是 Google 设计的协议，HTTP/3 就是跑在 QUIC 上的。它在 UDP 之上实现了：

- **前向纠错（FEC）**：发数据时附带冗余信息，少量丢包可以原地恢复，不用重传
- **连接迁移**：切换网络（WiFi → 4G）不断连
- **0-RTT 握手**：重连时省去握手延迟
- **多路复用**：一个连接承载多个流，避免队头阻塞

**Hysteria 2** 是基于 QUIC 的一个魔改版，专门针对 GFW 和晚高峰做了优化：

| 特性 | 传统 TCP (SS/VMess) | Hysteria 2 (QUIC) |
|------|-------------------|-------------------|
| 传输层 | TCP | UDP |
| 丢包处理 | 降速 + 重传 | FEC + 激进重传 |
| 握手 | TCP 三次 + TLS | QUIC 0-RTT |
| 队头阻塞 | 有 | 无（多路复用） |
| 拥塞控制 | Cubic/BBR | 自定义 Brutal |
| 伪装 | TLS | QUIC（天然像 HTTP/3） |

### 3.3 Brutal 拥塞控制算法

Hysteria 2 最核心的创新是 **Brutal** 算法。它的行为符合它的名字：

```
传统 TCP：   "堵车了？那我慢点开"
Brutal：    "堵车了？我走应急车道"
```

Brutal 不根据丢包来调整速率。你给它设定一个目标带宽（比如 50Mbps），它就恒定以这个速率发送，不管丢了多少包。丢失的数据通过 FEC 和激进的 retry 来补。

> ⚠️ **风险提示**：这确实很"粗暴"。如果运营商有深度包检测（DPI），大量异常 UDP 流量可能触发 QoS 限速甚至临时阻断。

## 四、实测对比

我们用同一台 RackNerd 洛杉矶 VPS，分别在白天和晚高峰测试：

### 白天（14:00）

| 协议 | 下载速度 | YouTube 画质 | 延迟 |
|------|---------|-------------|------|
| Shadowsocks (TCP) | 85 Mbps | 4K 秒开 | 160ms |
| Hysteria 2 (UDP) | 90 Mbps | 4K 秒开 | 158ms |

白天两者差距不大，TCP 的良好链路下表现不差。

### 晚高峰（21:30）

| 协议 | 下载速度 | YouTube 画质 | 延迟 |
|------|---------|-------------|------|
| Shadowsocks (TCP) | 2.3 Mbps | 720p 缓冲 | 360ms |
| Hysteria 2 (UDP) | 52 Mbps | 4K 秒开 | 175ms |

**晚高峰下 Hy2 速度快了 22 倍。** 这不是魔法——只是 TCP 在 12% 丢包率下已经崩溃了，而 QUIC/UDP 完全不受影响。

## 五、部署

追求极简部署，一键就能跑起来。

### 5.1 服务端（VPS）

```bash
# 一键脚本
bash <(curl -fsSL https://raw.githubusercontent.com/Misaka-blog/hysteria-install/main/hy2/hysteria.sh)
```

选 `[1] 安装 Hysteria 2` → 自签证书 → 设置端口（如 54321）→ 记住输出的 `hysteria2://` 链接。

### 5.2 客户端

**Windows**：v2rayN v6.0+，Ctrl+V 粘贴 `hysteria2://` 链接  
**Android**：NekoBox，自动识别导入  
**iOS**：Shadowrocket，类型选 Hysteria2，开启"允许不安全连接"  
**macOS/Linux**：使用 `sing-box` 或 `hysteria` CLI

## 六、最佳实践：双协议共存

我不建议只用 Hy2。原因：

1. **UDP QoS**：部分运营商（广东电信、某些校园网）对 UDP 流量做严格限制，Hy2 可能直接被限到 1Mbps
2. **断流风险**：异常 UDP 流量可能被运营商的 DPI 临时阻断 5-10 分钟
3. **耗电**：手机端 UDP 持续传输比 TCP 费电

**推荐配置**：

```
白天 / 需要省电      → Shadowsocks (TCP)
晚高峰 / 需要速度    → Hysteria 2 (UDP)
极端情况 / Hy2 被 QoS → Shadowsocks 作为保底
```

在 v2rayN 里把两个节点都加进去，右键设置 `自动切换` 或者手动一键切换。

## 七、总结

本文想传达的核心思想是：

> **不是你的 VPS 不够快，是 TCP 协议在丢包场景下不适合你。**

理解了这一点，你就理解了为什么 Hy2 能"拯救垃圾线路"，也理解了它的风险所在。技术没有银弹，只有正确的场景。
